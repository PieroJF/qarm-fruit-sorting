### Lab fixes
- `survey_capture.py:29` — `_SETTLE_JOINT_TOL_RAD` 0.015 → 0.04. The previous 0.015 rad joint-norm gate was tighter than the QArm position-mode PID's steady-state error (~0.025 rad) at the current `survey1` pose, so every capture aborted mid-batch. The downstream chessboard residual check (3 mm warn, 10 mm error) remains the real mm-level guard.

### Thread 1 — verify tomato detection in lab (highest priority)

The previous session ended with `[picker] no tomato visible` reports from the live picker. Investigation got as far as confirming the **detector contour gates accept the tomato** (`diag_why_no_tomato.py` showed contours #10 area=31255 and #11 area=37590 both "ACCEPT as tomato"), but `diag_detector.py` reported `0 detection(s)` — which we now know was the buggy `cam_extrinsics_survey1` drop, not a real detection failure. The diag is fixed but **was not re-run on the same scene before the lab session ended**.

**First action next lab session:**
1. Place a tomato + a banana under `survey1`.
2. Run `py -3.13 python/diag_detector.py`.
3. Read the `N detection(s)` line and open `logs/diag_detector_annotated.png`. Tomato class draws a red box (BGR `(0, 0, 255)`).
4. **If a tomato detection appears** → production pipeline detects tomatoes; the original "no tomato visible" was almost certainly because no tomato was actually in view at that moment. Move on.
5. **If still 0 tomato detections** → real failure downstream of the contour gates. Add temporary `print` calls inside `fruit_detector.detect_fruits` around the `pixel_to_base_frame` call to identify which contour fails projection (the `try / except ValueError: continue` is silently dropping them). Most likely culprit: `lam <= 0` in the ray-plane intersection if the survey1 extrinsic geometry is off.

### Thread 2 — strawberry-box (`plastic_box`) detection

The user wants both `strawberry` (single fruit, current `s` key) and `plastic_box` (whole Aldi clamshell, would map to a new `p` key) detectable simultaneously. Brainstorm paused at design Q2 (wrist-orientation strategy). Three options on the table:

- **A.** Operator manually orients the box; arm does default-orientation pick (smallest change).
- **B.** Detector computes box long-edge angle; controller rotates wrist via `gripper_mount_offset_rad` calibration (matches original spec's P2; large change).
- **C.** Ship A first to validate detection/grasp, defer B to a follow-up spec (recommended given deadline).

After the user picks one, the brainstorming flow continues with: basket teach point (`teach_points.placebox`), HSV/Canny tuning for the box, then writing the spec at `docs/superpowers/specs/2026-04-27-strawberry-and-box-coexist-design.md`.

The original (now-superseded) plan that proposed REPLACING strawberry with plastic_box lives at `docs/superpowers/specs/2026-04-24-accurate-pick-and-box-swap-design.md` — its P0 (solvePnP rewrite + capture settle) is implemented; P1 (plastic_box swap) is what we're rewriting to keep both classes.
